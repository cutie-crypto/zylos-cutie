```
$ ssh kolzy-134 'pm2 list'

$ ssh kolzy-134 'cat /home/kolzy/zylos/.claude/skills/cutie/package.json | grep version'

```

`restart=32` 是历史 selfUpgrade 期间反复 PM2 重启的累积，本次新版本起来后未再增长（uptime 稳定 27m）。

### 2. selfUpgrade 循环已停止

最后一次 "Upgrade available" 出现在 06:03:48（升级 2.0.0 周期内）。06:03:55 进程起来之后，server 不再下发升级，证明 server 已识别 `connector_version=2.0.1` 与 `target_version=2.0.0` 不匹配的"≥ target"语义（或直接接受 2.0.1）。

### 3. B4 透传到 prompt（kol_user_id 真实落点验证）

直接在 kolzy 上调 `buildPrompt`：

```js
buildPrompt({
  message: "测试用户消息",
  kol_user_id: "kol_real_12345",
  caller_user_id: "user_real_67890",
  scene: "app_kol_ask",
})
```

输出片段：

```
# CONTEXT
kol_user_id=kol_real_12345
caller_user_id=user_real_67890
scene=app_kol_ask

# USER
测试用户消息
```

断言全过：
- contains `kol_real_12345`: true
- contains `user_real_67890`: true
- contains `app_kol_ask`: true
- 不再包含 hardcode `kol_user_id=unknown`: true

**间接证明完整链路**：`buildPrompt` 对 falsy `kol_user_id` 直接 `throw new Error('buildPrompt: kol_user_id required')`。06:11:38 task 12.8s 成功完成 → 必然意味着 server → connector dispatcher → adapter.callAgent → runner → buildPrompt 全链路 kol_user_id 真实透传，否则 throw 会让 task 立即失败。

## 模式观察

1. **kolzy ≠ COCO** — kolzy 是我们能 ssh 的同形态机器，可观测 stderr / version 上报 / cold-start 复现；COCO 是 Zylos 的机器，纯黑盒，只能从 App 端看返回。需要日志的验证统统在 kolzy 做（已记入 memory: `project_coco_blackbox.md`）
2. **stale dist 事故的结构性防御已就位** — `package.json.scripts.prepublishOnly` 改 publish 必跑 build + test；后续避免 `npm publish` 前忘 `npm run build` 的事故复发
3. **buildPrompt strict assertion 的诊断价值** — 强制 `kol_user_id` 必传 + 直接 throw，让 B4 透传不通时立即可见，不会"静默用 'unknown' 跑通"。这种"硬错优于软降级"在 review 阶段曾被列为 silent-failure 修复，证明价值
4. **server 端 `error_type` 实为自由文本** — 本次 review 中纠正了 BACKLOG #7 注释里"server CHECK 约束只接受 openclaw_error"的误判。VARCHAR(50) 无 CHECK，业务层零 enum 判断 → wire 上压平 'openclaw_error' 是过度防御，可以解掉
